### Key Mental Model Shifts:

**1. Type Awareness:**
```java
// Python: Flexible, runtime type checking
x = 5
x = "now a string"  # Valid

// Java: Strict, compile-time type checking
int x = 5;
x = "now a string";  // COMPILE ERROR - type safety
```

**2. Explicit Structure:**
```java
// Python: Quick and loose
def calculate(a, b):
    return a + b

// Java: Formal and structured
public class Calculator {
    public int calculate(int a, int b) {
        return a + b;
    }
}
```

**3. Verbosity vs Clarity:**
- Java requires more code, but provides better IDE support
- Type declarations prevent runtime errors
- Compile-time checking catches bugs early

## üìö KEY CODE PATTERNS LEARNED

### Pattern 1: Class with Encapsulation (Most Important for Selenium POM)

```java
// This pattern is fundamental for Page Object Model in Selenium
public class LoginPage {
    // Private instance variables (encapsulation)
    private String username;
    private String password;

    // Constructor
    public LoginPage(String username, String password) {
        this.username = username;
        this.password = password;
    }

    // Getters and Setters
    public String getUsername() {
        return username;
    }

    public void setUsername(String username) {
        this.username = username;
    }

    // Business logic method
    public boolean authenticate() {
        // Validation logic
        return username != null && password != null;
    }
}
```

**Why it matters for SDET:** This is the exact pattern used in Page Object Model for Selenium.

### Pattern 2: Collections with Generics (Critical for Test Data)

```java
// ArrayList - dynamic array (like Python list)
List<String> testUsers = new ArrayList<>();
testUsers.add("user1@test.com");
testUsers.add("user2@test.com");

// HashMap - key-value pairs (like Python dict)
Map<String, String> credentials = new HashMap<>();
credentials.put("admin", "admin123");
credentials.put("user", "user123");

// Iterating through collections
for (String user : testUsers) {
    System.out.println(user);
}

// HashMap iteration
for (Map.Entry<String, String> entry : credentials.entrySet()) {
    System.out.println("Username: " + entry.getKey() + ", Password: " + entry.getValue());
}
```

**Why it matters for SDET:** Test data management, storing test results, managing multiple test scenarios.

### Pattern 3: Exception Handling (Essential for Test Stability)

```java
public void performAction() {
    try {
        // Code that might throw exception
        int result = 10 / 0;  // Division by zero
    } catch (ArithmeticException e) {
        // Handle specific exception
        System.out.println("Error: " + e.getMessage());
        // Log error, take screenshot in Selenium
    } catch (Exception e) {
        // Handle any other exception
        System.out.println("Unexpected error: " + e.getMessage());
    } finally {
        // Always executes (cleanup code)
        System.out.println("Cleanup complete");
        // Close browser, release resources
    }
}
```

**Why it matters for SDET:** Handling flaky tests, network issues, element not found errors.

---

### Pattern 4: Inheritance Hierarchy (Framework Base Classes)

```java
// Base class (like BaseTest in Selenium frameworks)
public class Vehicle {
    protected String brand;  // Accessible to subclasses
    protected int year;

    public Vehicle(String brand, int year) {
        this.brand = brand;
        this.year = year;
    }

    public void start() {
        System.out.println(brand + " is starting...");
    }
}

// Subclass (like specific page classes extending BasePage)
public class Car extends Vehicle {
    private int doors;

    public Car(String brand, int year, int doors) {
        super(brand, year);  // Call parent constructor
        this.doors = doors;
    }

    @Override
    public void start() {
        super.start();  // Call parent method
        System.out.println("Car with " + doors + " doors ready.");
    }
}
```

**Why it matters for SDET:** BasePage, BaseTest patterns in frameworks. Code reusability.

---

### Pattern 5: Interface Implementation (Flexible Design)

```java
// Interface - contract for behavior
public interface Searchable {
    void search(String query);
    List<String> getResults();
}

// Implementation
public class ProductSearch implements Searchable {
    private List<String> results = new ArrayList<>();

    @Override
    public void search(String query) {
        // Search logic
        results.add("Product matching: " + query);
    }

    @Override
    public List<String> getResults() {
        return results;
    }
}

// Usage with polymorphism
Searchable search = new ProductSearch();
search.search("laptop");
List<String> results = search.getResults();
```

**Why it matters for SDET:** Abstraction in frameworks, swappable components (different browsers, different environments).

---

### Pattern 6: String Manipulation (Test Data Processing)

```java
public class StringUtilities {

    // String comparison (CRITICAL - never use ==)
    public static boolean isValid(String input) {
        return input != null && !input.isEmpty() && !input.trim().equals("");
    }

    // String building (efficient for concatenation)
    public static String buildTestReport(List<String> testNames) {
        StringBuilder report = new StringBuilder();
        report.append("Test Report\n");
        report.append("===========\n");
        for (String test : testNames) {
            report.append("- ").append(test).append("\n");
        }
        return report.toString();
    }

    // String parsing
    public static String[] parseCSV(String line) {
        return line.split(",");
    }
}
```

**Why it matters for SDET:** Processing test data, parsing CSV/Excel files, building reports.

---

### Pattern 7: Method Overloading (Flexible Test Methods)

```java
public class TestUtility {

    // Same method name, different parameters
    public void login(String username, String password) {
        System.out.println("Login with username and password");
    }

    public void login(String email) {
        System.out.println("Login with email only (passwordless)");
    }

    public void login() {
        System.out.println("Login with default credentials");
    }

    // Usage
    public void runTests() {
        login("user", "pass");  // Calls first method
        login("user@test.com");  // Calls second method
        login();  // Calls third method
    }
}
```

**Why it matters for SDET:** Creating flexible test utility methods, handling different test scenarios.
